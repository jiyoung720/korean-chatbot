# 멘토링 노트

부트캠프 강사님과의 멘토링/수업에서 받은 조언을 기록합니다. 우리가
직접 실험한 내용(`training_log.md`)과는 별도로, 외부에서 받은 피드백과
앞으로의 방향을 정리하는 용도입니다.

## 2026-06-24

### 질문: 결과물보다 과정/디버깅 경험이 포트폴리오로 의미가 있는가

답변 요지: 의미 있다. 다만 다음 두 가지를 더하면 더 강해진다.

- **스토리텔링을 구체적인 수치로**: "잘 안 됐다"가 아니라 "몇 %를
  기대했는데 실제로 몇 %밖에 안 나왔다"처럼 정량화할 것
- **프론티어 모델과 직접 비교**: GPT-4/Claude 등에 같은 작업을 시켜서
  나온 결과와 비교해 몇 점인지 평가해볼 것

### 두 가지 발전 방향

1. **작은 모델을 계속 개선**: 프론티어 모델 API로 합성 데이터셋을
   만들어 직접 만든 작은 모델을 fine-tuning(knowledge distillation)
2. **도메인을 좁히기**: "평균 모델급 응답 일관성"이 목표였지만 자원/
   데이터 부족으로 무리일 수 있음 — 특정 분야로 좁혀서 그 안에서만
   확실한 티카타카가 되는 모델로 목표 수정도 고려 가능

### 가장 중요한 메시지: 엔진 vs 에이전트

> "취업에서 요구하는 건 기능을 하는 에이전트를 만드는 것과 서빙임.
> 지금 개발하는 건 엔진을 만드는 것임. 자기 모델을 만드는 거 좋은
> 시도지만 에이전트와 서빙도 잊지 말아야 함."

- 직접 GPT를 구현하는 것 = "엔진 만들기" 역량 (연구/구현)
- 기성 LLM으로 실제 동작하는 서비스를 만들고 배포하는 것 = "에이전트/
  서빙" 역량 (실무)
- 신입 채용 시장은 후자를 더 많이 요구함 → 둘 다 보여줄 필요

**결론**: 메인 프로젝트(직접 구현한 GPT)는 API를 섞지 않고 그대로
유지. 대신 강사님이 이미 내주신 미니 프로젝트(Gemini API 또는 Qwen/
Gemma + RAG + FastAPI + RAGAS)를 별도 프로젝트로 진행해 "에이전트/
서빙" 역량을 채운다. 두 프로젝트는 README에서 "서로 다른 목적"이라고
명확히 구분해 포트폴리오에 나란히 둔다.

이번 주~다음 주 중 강사님과 메인 프로젝트 진행 상황을 정리해 다시
논의하기로 함. 도메인을 좁히는 것 등 메인 프로젝트의 큰 방향 수정은
이 논의 이후에 결정.

## 2026-06-24 (수업 - 평가/관찰 관련)

### 모델 평가 방법

- 자기 모델(컨텍스트 윈도우가 작음, 우리 모델은 max_seq_len=256 —
  강사님이 말한 "커야 1024"보다도 작음) / 공개 가중치 모델(Qwen,
  Gemma 등) / 프론티어 모델(Gemini, OpenAI, Anthropic)을 **동일한
  평가 기준**으로 비교
- 비교하려면 개인 프로젝트에 RAG 시스템이 있어야 함
- **검색(retrieval)을 끄고 모델만 비교하는 것을 권장**:
  ```python
  chain = prompt | model | parser  # retriever 없이 모델 자체만 평가
  ```
  검색 품질과 생성 품질을 분리해서 보기 위함. 우리가 training_log에
  "검색은 정확한데 생성이 병목"이라고 분석한 것을, 검색을 완전히
  제거한 형태로 한 번 더 검증해볼 수 있음.
- 자기 모델이 없어도(API 모델끼리만) 비교 평가해보는 것도 권장됨

### LLM을 judge로 쓸 때

- 사람이 평가하거나 AI가 평가하거나 둘 다 가능
- AI에게 평가시킬 때 **"해당 분야의 전문가" 페르소나**를 주는 것이
  평가 품질에 도움이 됨

### LangSmith에 대한 입장

- **관찰(Tracing)은 해볼 것을 권장 (필수)**
- 평가(evaluation)는 선택 — 해도 되고 안 해도 됨
- **프롬프트 관리 기능은 지양** — 프롬프트를 코드가 아닌 것으로
  관리하는 데는 한계가 있음 (비개발자가 프롬프트를 작성해도 결국
  개발자가 손을 대게 되므로, 비개발자 직접 관리라는 전제 자체가
  현실과 안 맞는 경우가 많음)
- LangSmith는 lock-in 위험이 있을 수 있으나, 대체 가능한 관찰 도구가
  많아 의존성이 크지 않음. 안 쓰게 되더라도 바로 꺼도 문제 없음.
  "좋은 점이 있지만 정답은 아님"

### 다음 LangSmith 작업 시 적용할 것

1. Tracing(관찰)은 확실히 해보기
2. Dataset 기반 평가는 선택적으로, 너무 깊게 파지 않기
3. 프롬프트 관리 기능은 사용하지 않기
4. 평가 실험에 "RAG 없이 모델만" 비교하는 케이스를 포함해, 컨텍스트
   윈도우 한계와 생성 능력 한계를 분리해서 보여주기

## 2026-06-24 (Slack, 1:N 멘토링 준비)

### 상담 전 질문: 컨텍스트 윈도우(256)를 미리 늘리고 가야 할지

**질문**: 오늘 수업 때 강사님이 "본인이 직접 만든 챗봇의 컨텍스트
윈도우가 최소 1024 정도 될 것이다"라고 하셨는데, 우리는 학습 시간과
데이터 길이(위키 문서, 대화 데이터)만 보고 256으로 정했다. 이걸 미리
키우고 상담에 가야 하는지 질문.

**답변(Alex)**: "256... 일단 그 길이를 감안하고 평가 데이터셋을
마련해보세요."

→ 지금 재학습으로 컨텍스트 윈도우를 늘리는 게 우선순위가 아님.
256이라는 제약을 그대로 받아들이고, 그 안에서 평가할 수 있는
데이터셋을 만드는 것이 다음 행동.

### 상담 준비 방향 확인

**질문**: 1:N 멘토링 때 작업 내용을 정리해 오라고 하셨는데, "어떤
학습을 시켰고, 만들고 있는 과정과 결과, 결과물의 한계, 앞으로의
방향성"으로 준비하면 맞는지.

**답변(Alex)**: 맞다 — 정리하면:
- 어떻게 학습 시켰고
- 결과
- 한계점
- 방향성

### 다음 할 일: 공통 평가 시스템 구축

앞으로의 비교 분석을 위해 다음 세 가지를 **공통 기준으로 평가할 수
있는 평가 시스템**을 만들 것:

1. **엘라 모델** — 직접 만든 모델(`korean-chatbot`의 wiki/dialog 모델)
2. **Gemma 4 e2b** — 공개 가중치 모델 (맥에서 로컬 실행 가능)
3. **Gemini 2.5 Flash** — 프론티어 모델 API

**평가 데이터셋 만드는 법**: "이런 질문에 답을 잘 하면 좋겠다"는
질문과 모범 답을 직접 만들어 평가 데이터셋으로 구성. **항목은 10개부터
시작**.

예시 형태:
```
질문: "세종대왕은 누구야?"
모범 답: "세종대왕은 조선의 제4대 왕으로, 1443년 한글(훈민정음)을 창제했다."
```
