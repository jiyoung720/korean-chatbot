# 회고: LangChain RAG와 평가 시스템 구축

이 글은 Vanilla RAG를 완성한 이후부터, LangChain 기반 RAG 구축과 
평가 시스템까지의 과정에서 무엇을 배웠는지를 정리한 것이다.

## 처음 생각: LangChain을 쓰면 좀 더 나아지려나?

Vanilla RAG를 직접 구현한 후에는 한 가지 의문이 남았다. 
내가 직접 만든 RAG는 Retriever와 프롬프트 조립을 다 하나하나 하는데,
실제 프로젝트들은 왜 LangChain 같은 프레임워크를 쓰는 걸까? 

실무에서 LangChain을 써먹을 수 있다는 말씀을 듣고, 이걸 적용시키면
결과가 달라지려나 하는 생각도 좀 있었다.

## LangChain 마이그레이션: 생각보다 별로 나아지지 않았다

같은 임베딩 모델(`jhgan/ko-sroberta-multitask`), 같은 코퍼스(위키 1,000개 문서),
같은 생성 모델(Ella 23M)으로 Vanilla RAG와 LangChain RAG를 구성해서 
"세종대왕은 누구야?"를 테스트했다. 

결과적으로 거의 같은 답변이 나왔다.

같은 임베딩 모델, 같은 코퍼스, 같은 생성 모델을 사용하면 검색 결과와 최종 응답은 본질적으로 
거의 동일했다. 결국 생성 품질은 모델 자체가 중요하지 프레임워크와 크게 관계는 없었던 것 같다.

## FAISS와 Chroma

LangChain 마이그레이션 과정에서 또 다른 배움이 있었다. 
처음엔 FAISS를 쓰려고 했는데 macOS ARM에서 segmentation fault가 났다.
환경 이슈여서 FAISS를 쓰기 어려운 것 같았다.
그래서 Chroma로 바꿨는데, 여기서도 에러가 났다. 기본 거리 함수가 L2여서
코사인 유사도 기반이던 Vanilla RAG와 검색 결과가 달라졌다.

```python
# 해결책: Chroma 생성 시 명시적으로 코사인 설정
build_or_load_vectorstore(
    collection_name="wiki",
    collection_metadata={"hnsw:space": "cosine"}  # 이 줄 수정
)
```

라이브러리 사용 시에도 어떤 환경에서 어떤 설정을 해야 돌아가는지를 잘 알아야겠다고 생각했다.

## 평가

LangChain RAG가 완성되고 나서 지금 내가 진행하고 있는 프로젝트의 챗봇의 수준을
판단할 수 있도록 judge를 해보라고 하셨다

그래서 `evaluation_dataset_v1.md`를 만들어 10개 질문과 모범답안을 정의하고,
Ella(내가 만든 모델), Gemma 4 8B, Gemini 2.5 Flash 세 모델의 응답을 비교했다.

## 세 모델 비교: 메타인지 능력의 차이가 나타났다

나의 모델의 결과가 낮은 점수를 받은 건 놀랍지 않았다...(ㅠㅠ)

흥미로운 결과는 검색 실패 사례(Q10)에서 나왔다.
모든 모델이 같은 Retriever를 써서 같은 검색 결과를 받았는데,
생성 단계에서 다른 대응을 보였다:

- **Ella**: 무관한 인물 사망 목록을 그대로 생성했다 (hallucination)
- **Gemma**: "제공된 정보에는 세종대왕에 대한 내용이 없습니다" (정보 부족 인지)
- **Gemini**: "모른다." (정보 부족 인지)

"생성 모델의 크기가 크다 = 답변이 더 자연스럽다" 정도의 차이만 예상했는데,
실제로는 **"주어진 정보가 부족할 때 그 사실을 인식하는 능력"** 같은 
메타인지 능력의 차이가 명확하게 드러났다. 

이게 정말 중요한 발견이라고 느꼈는데 그 이유가 모르는데 이상한 답을 뿜어내는건 사용자 입장에서
매우 위험하기 때문이다. 나도 AI 한테 질문을 했을 때, 잘못된 정보를 정중하게 그럴듯하게 지어내며
"모르겠습니다"라고 정직하게 답하지 않으면 큰 혼란에 빠졌던 경험이 있었기 때문에 더 그렇게 느꼈다.

## LLM-as-a-Judge: 숫자로 표현

정성적 비교뿐 아니라 정량적으로도 평가하였다.
Gemini를 Judge로 삼아 정확성, 질문 관련성, 자연스러움 세 기준으로
자동 채점을 했다. 그 결과:

- Ella: 1.23점 / Gemma: 4.57점 / Gemini: 4.00점 (각 5점 만점)

그동안 느낌으로 알고 있던 차이를 숫자로도 확인할 수 있었다.
다만 한계도 명확했다 — Judge 모델(Gemini)이 평가 대상(Gemini)과 일부 동일하고,
10개 질문이라는 작은 샘플이라는 점이다.

## LangSmith Evaluation: 평가를 체계화

LLM-as-a-Judge까지는 Jupyter 노트북 수준이었는데, 
이번에는 한 단계 더 나아가 LangSmith라는 운영 도구로 평가를 통합했다.

프로세스:
1. `build_dataset.py`: Evaluation v1 데이터셋을 LangSmith Dataset으로 등록
2. `run_evaluation.py`: Gemini Judge를 이용해 자동 채점
3. LangSmith UI에서 결과를 대시보드처럼 확인

특별히 신경 썼던 부분은 **Q10에 메타데이터를 붙이는 것**이었다.
```python
{
  "known_retrieval_issue": True,
  "note": "주어 추출 정규식이 '어떤 ~로 유명한가' 패턴을 처리하지 못해..."
}
```

이렇게 하면 나중에 "Q10이 왜 0점을 받았나?"를 볼 때
"이게 모델이 못해서인가, 아니면 파이프라인 버그인가"를 구분할 수 있다.
평가 결과가 커질수록 맥락의 정보가 중요하다는 것도 배웠다.

## 가장 크게 달라진 점

처음에는 "모델을 구현하고 성능을 높인다"에 집중했다.
이번 단계에서는 "실험을 설계하고, 결과를 비교하고, 평가를 체계화한다"는
새로운 층위의 작업을 경험했다.

모델 구현도 중요하지만, 그 모델이 정말 어느 정도로 좋은지 
누구나 재현 가능한 방식으로 증명하는 것도 AI 프로젝트에서 
못지않게 중요하다는 걸 배웠다.
